## 1. 목적

이 문서는 MFC 기반 도면Pro를 웹 기반 시스템으로 재구축하기 위한 전체 작업을 우선순위, 선후 의존성, 산출물과 완료 기준에 따라 목록화한 실행 계획서다.

다음 목적을 가진다.

- 프로젝트 전체 범위를 누락 없이 관리한다.
- 먼저 해결해야 하는 기술·데이터·업무 위험을 식별한다.
- 각 작업의 착수 순서와 선행 조건을 명확히 한다.
- 작업을 작은 구현 단위로 분해할 수 있는 공통 기준을 제공한다.
- 구현, 검증, 배포, 문서화까지 포함해야 작업이 완료된 것으로 판단한다.
- 각 작업에 착수할 때 [세부 작업계획 템플릿](./work-item-template.md)을 사용해 실행 단계를 구체화한다.

이 문서는 캘린더 일정표가 아니다. 팀 구성과 가용 인력이 확정되기 전에는 임의의 완료 날짜를 약속하지 않고, 우선순위와 의존성 기반으로 실행 순서를 관리한다. 인력과 업무 검수 가능 시간이 확정되면 이 작업 항목을 스프린트 또는 마일스톤 일정으로 배치한다.

## 2. 확정 범위와 전제

| 항목 | 기준 |
|---|---|
| 최종 목표 | MFC 프로그램의 범위 내 업무를 웹 기반 시스템으로 재구축하고 검증 후 MFC 신규 입력을 종료한다. |
| 입면도 | 구현하지 않는다. 화면, 메뉴, 편집기, API, 활성 데이터 모델, 기능 데이터 이전 및 테스트 범위에서 제외한다. |
| 기계 연동 | 최종 범위에는 포함한다. 1단계에는 비활성 항목, 권한, 상태 및 계약 자리만 두고 실제 Agent와 통신은 후속 단계에서 구현한다. |
| 인증 | MFC 실행·로그인 방식과 독립된 웹서비스 전용 인증·세션·RBAC를 구축한다. |
| 운영 DB | RDS 또는 운영 서버에 별도 PostgreSQL을 구축해 웹의 단일 원본으로 사용한다. |
| 개발·테스트 DB | 로컬 PostgreSQL을 사용한다. SQLite 대체 테스트는 사용하지 않는다. |
| DB 설계·코드 | Prisma Schema, Prisma Client, Prisma Migrate를 표준으로 사용한다. |
| 언어 | 1차 완료 범위는 대한민국·한국어 단일 언어이며 다국어는 후속 범위다. |
| 웹 구조 | Next.js 기반 모듈형 모놀리스로 시작하고 긴 작업은 worker로 분리한다. |
| 레거시 검증 | MFC 입력·결과를 golden master로 자동 대조하되 MFC를 절대 정답으로 보지 않는다. 레거시 오류, 웹 개선과 업무 규칙 변경을 분류·승인해 별도 기대값으로 검증한다. |
| 전환 방식 | 모듈별 검증과 병행 운영을 거치는 점진적 전환으로 진행한다. |

MFC 참조 경로의 입면도 코드는 레거시 업무 흐름을 이해하고 원본을 보존하는 목적으로만 본다. 입면도 기능을 웹 backlog에 다시 넣으려면 별도의 범위 변경 승인과 일정 재산정이 필요하다.

## 3. 우선순위 정의

우선순위는 기능의 가치만이 아니라 선행 의존성, 실패 위험과 전체 일정에 미치는 영향을 함께 반영한다.

| 우선순위 | 의미 | 처리 원칙 |
|---|---|---|
| `P0` | 착수 차단 요소 또는 프로젝트 전체의 불확실성을 결정하는 작업 | 다른 기능 구현보다 먼저 시작한다. 완료 또는 명시적 결정 없이는 의존 작업에 착수하지 않는다. |
| `P1` | 웹 시스템의 핵심 업무를 사용할 수 있게 만드는 필수 작업 | P0 기반이 준비되는 즉시 진행한다. 핵심 절곡·저장·수주 흐름을 구성한다. |
| `P2` | 생산 운영, 이전, 출력, 보안과 운영 완성도를 확보하는 작업 | 핵심 업무 계약이 안정된 뒤 진행하되 데이터 조사와 외부 Engine 비계승·대체 전략은 조기에 확정한다. |
| `P3` | 선행 시스템 안정화 후 수행하는 후속 작업 | 중요도가 낮다는 뜻이 아니다. 기계 연동과 최종 전환처럼 앞 단계 결과에 의존하므로 뒤에 수행한다. |

긴급 요청이 들어와도 P0/P1의 데이터 무결성, 계산 타당성, 차이 판정, 보안 및 복구 기준을 생략하지 않는다.

## 4. 전체 실행 흐름과 완료 게이트

```mermaid
flowchart LR
  G0["G0 범위·기준선 확정"] --> G1["G1 PostgreSQL·Prisma 기반 완성"]
  G1 --> G2["G2 절곡 핵심 검증·개선"]
  G2 --> G3["G3 수주 업무 완성"]
  G3 --> G4["G4 생산·출력 준비"]
  G4 --> G5["G5 데이터 이전·운영 검증"]
  G5 --> G6["G6 기계 연동 검증"]
  G6 --> G7["G7 병행 운영·MFC 종료"]
```

| 게이트 | 통과 조건 |
|---|---|
| `G0` 기준선 | 범위, 실제 운영 데이터 원본, 핵심 MFC 흐름, 외부 엔진 보유 상태와 golden master 확보 계획이 승인됨 |
| `G1` 플랫폼 | 로컬·운영 PostgreSQL 정책, Prisma migration, 인증·권한·감사, CI와 백업·복구 기본 경로가 검증됨 |
| `G2` 절곡 | MFC 비교 차이가 유형별로 판정·승인되고, 유지 대상은 허용 오차 내 일치하며 레거시 오류 수정은 승인 기대값과 회귀 테스트를 보유함 |
| `G3` 수주 | 거래처부터 절곡 작업 스냅샷, 계산, 가격, 승인까지 실제 수주 흐름을 처리할 수 있음 |
| `G4` 생산 | 재단, 파일 생성, PDF·라벨, 비동기 작업의 실패·재시도와 현장 출력이 검증됨 |
| `G5` 이전·운영 | 반복 가능한 데이터 이전, 대조 보고, 모니터링, 보안, 백업·복구 및 UAT가 승인됨 |
| `G6` 기계 | Agent, 장비별 통신, 중복 전송 방지, 장애 복구와 현장 시험이 완료됨 |
| `G7` 전환 | 최종 이전과 병행 운영을 통과하고 웹 신규 입력 전환 및 MFC 읽기 전용 보관이 완료됨 |

게이트가 통과되지 않은 상태에서도 독립적인 조사나 프로토타입은 진행할 수 있다. 다만 해당 게이트에 의존하는 운영 기능을 완료로 선언하거나 운영 데이터에 적용할 수는 없다.

## 6. 권장 착수 순서

처음부터 모든 기능을 동시에 구현하지 않는다. 다음 순서로 첫 실행 묶음을 시작한다.

1. `P0-01` 범위 기준선과 변경관리 규칙을 확정한다.
2. `P0-02`~`P0-04`에서 MFC 기능과 실제 운영 데이터 원본을 조사한다.
3. `P0-06` golden master 20건을 수집하며 계산 결과와 허용 오차 논의를 시작한다.
4. `P0-08`에서 운영·로컬 PostgreSQL 환경 기준을 확정한다.
5. `P0-09`~`P0-12`에서 Prisma Schema, migration, 테스트 DB와 CI 기반을 만든다.
6. `P1-01`~`P1-04`에서 인증·권한·감사와 기계 연동 placeholder를 만든다.
7. `P1-05`~`P1-08`에서 절곡 문서 계약, 저장, 템플릿 개정을 완성한다.
8. `P1-09`~`P1-17`에서 MFC 차이를 판정하며 계산·편집·전개·DXF의 업무 타당성과 웹 사용성을 확보한다.
9. 수주와 생산 기능은 각각 G2와 G3를 통과한 계약을 기반으로 확장한다.
10. 기계 연동 실제 구현은 G5 이후 별도 작업 묶음으로 착수한다.

초기 개발의 첫 번째 수직 절단은 다음 흐름으로 잡는다.

```text
로컬 PostgreSQL
→ Prisma migration
→ 사용자 로그인
→ 절곡 초안 생성·저장
→ 새로고침 후 복원
→ 계산 preview
→ 감사 로그 확인
```

이 흐름이 통과하면 DB, 인증, 문서 저장, 계산 패키지와 UI가 실제로 연결된 최소 기반을 확보한 것으로 본다.

## 7. 작업 착수 시 세부계획 수립 규칙

각 작업 ID에 착수하기 전 `docs/work-items/` 아래에 별도 작업 문서를 만든다.

권장 파일명:

```text
docs/work-items/P1-07-fold-draft-persistence.md
```

작업 문서는 [세부 작업계획 템플릿](./work-item-template.md)을 복사해 작성하고 다음 항목을 확정한 뒤 구현을 시작한다.

### 7.1 착수 준비

1. 작업 목적과 사용자 결과를 한 문장으로 정의한다.
2. 포함 범위와 제외 범위를 구분한다.
3. 선행 작업이 실제 완료됐는지 확인한다.
4. MFC 근거 파일, 레거시 테이블과 현재 웹 코드를 연결한다.
5. 미결정 사항과 결정 책임자를 기록한다.
6. 데이터·API·화면·권한·감사·파일 영향도를 조사한다.
7. 허용 오차와 완료 기준을 측정 가능한 표현으로 바꾼다.

### 7.2 설계

1. 사용자 흐름과 실패 흐름을 작성한다.
2. 도메인 규칙과 상태 전이를 정의한다.
3. Prisma 모델과 migration 영향을 설계한다.
4. API 요청·응답·오류·idempotency·권한을 정의한다.
5. UI 상태, 자동 저장, 충돌과 접근성을 정의한다.
6. 외부 엔진·파일·worker·기계 의존성을 adapter 계약으로 분리한다.
7. 보안·개인정보·감사·백업·rollback 영향을 검토한다.
8. 필요하면 ADR을 먼저 작성하고 승인한다.

### 7.3 구현 분해

하나의 작업을 다음 순서의 작고 검증 가능한 단위로 나눈다.

1. 계약과 runtime schema
2. Prisma schema와 migration
3. 도메인 규칙과 단위 테스트
4. repository와 application use case
5. API와 통합 테스트
6. UI와 상태 관리
7. E2E·시각·성능 검증
8. migration·배포·rollback
9. 운영 문서와 사용자 문서

모든 작업이 위 요소를 전부 필요로 하지는 않는다. 필요 없는 항목은 삭제하지 말고 `해당 없음`과 이유를 남긴다.

### 7.4 검증·종료

1. 자동 테스트와 수동 검수 결과를 기록한다.
2. MFC 비교 작업은 입력·중간값·최종값 차이와 `PARITY_REQUIRED`, `LEGACY_DEFECT`, `WEB_IMPROVEMENT`, `RULE_CHANGE`, `UNRESOLVED` 판정을 보고한다.
3. DB 변경은 빈 DB 적용과 기존 DB upgrade를 모두 시험한다.
4. 운영 영향이 있으면 배포·rollback과 관측 지표를 시험한다.
5. 완료 기준을 항목별로 증명하는 링크나 결과를 남긴다.
6. 잔여 위험과 후속 작업을 새 ID로 등록한다.
7. 관련 종합 문서와 사용자 문서를 갱신한다.

## 8. 공통 완료 정의

다음 조건을 만족해야 작업 상태를 `완료`로 변경한다.

- 승인된 범위와 완료 기준을 충족했다.
- 관련 도메인 규칙에 단위 테스트가 있다.
- DB 작업은 Prisma migration과 로컬 PostgreSQL 통합 테스트가 있다.
- 변경 API는 인증·권한·입력 검증·표준 오류·감사 요구를 충족한다.
- 사용자 흐름 변경은 핵심 E2E 또는 재현 가능한 수동 검수 기록이 있다.
- 계산·geometry 변경은 유형별로 승인된 golden master 또는 개선 기대값과 허용 오차를 통과했다.
- 비동기·파일·외부 연동은 재시도, 중복 요청과 실패 경로가 검증됐다.
- 운영 변경은 환경변수, migration, 배포와 rollback 절차가 문서화됐다.
- 사용자 영향과 알려진 제한이 문서에 반영됐다.
- 임시 코드, 우회 플래그와 미해결 TODO가 후속 작업 없이 남아 있지 않다.

## 9. 작업 상태와 관리 규칙

| 상태 | 의미 |
|---|---|
| `BACKLOG` | 범위에 등록됐으나 상세계획을 작성하지 않음 |
| `READY` | 선행 조건, 상세계획, 완료 기준과 담당자가 확정됨 |
| `IN_PROGRESS` | 구현 또는 조사가 진행 중 |
| `VERIFYING` | 자동 테스트, MFC 대조, 사용자 또는 운영 검수 중 |
| `BLOCKED` | 외부 결정·자료·권한 없이는 진행할 수 없음 |
| `DONE` | 공통 완료 정의와 개별 완료 기준을 모두 충족 |

관리 규칙:

- 동시에 진행하는 작업 수를 제한하고 완료되지 않은 작업을 장기간 쌓지 않는다.
- `BLOCKED` 작업은 원인, 필요한 결정, 책임자와 다음 확인일을 기록한다.
- 범위가 커지면 기존 작업 안에 숨기지 않고 새 작업 ID로 분리한다.
- Prisma migration, API 계약, 계산 규칙은 검토 없이 병합하지 않는다.
- 사용자 승인과 현장 검수가 필요한 작업은 개발 완료와 업무 완료를 분리해 추적한다.
- 입면도 기능 요청은 일반 결함이 아니라 범위 변경으로 처리한다.
- 1단계 기계 연동에서 실제 통신 요구가 나오면 P3 작업으로 분리한다.

## 12. 범위 밖 작업

다음은 현재 프로젝트 계획에 포함하지 않는다.

- 입면도 CAD 편집기
- 입면도 템플릿 라이브러리와 입면도 출력
- 입면도 활성 데이터 모델과 API
- `draww`, `drawxy`, `drawvari`, `selldraw*`를 웹 업무 엔터티로 변환하는 기능
- 완전한 오프라인 편집
- 외부 ERP·회계 시스템 연동
- 확정되지 않은 추가 기계 또는 프로토콜
- MFC 코드의 기계적인 1:1 웹 변환
- MFC HiStarter·command line DB 연결·로그인 메커니즘 재사용
- SQLite 업무·테스트·다국어 런타임
- 1차 완료 단계의 다국어 UI와 번역 데이터

입면도 레거시 원본의 백업·건수·checksum·보관 manifest 작성은 범위에 포함된다. 이는 기능 구현이나 활성 데이터 이전이 아니다.
