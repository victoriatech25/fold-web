## 3. 프로젝트 목표

MFC 도면Pro의 확정 대상 업무와 데이터 자산을 웹 기반 시스템으로 재구축하고, 레거시 비교·업무 타당성·현장 검증·데이터 이전·병행 운영을 거쳐 웹을 신규 업무의 단일 원본으로 전환한다.

MFC 기능을 코드 단위로 그대로 복제하는 것이 아니라 절곡, 기준정보, 수주, 계산, 생산, 출력과 기계 연동의 도메인 경계를 분리해 유지보수 가능한 웹 시스템으로 다시 설계한다.

MFC 코드, UI와 계산 결과는 중요한 분석·비교 자료지만 절대적인 정답으로 보지 않는다. 웹의 특성과 편의성을 반영해 업무 흐름을 재구성하고, MFC 계산 오류와 불합리한 구성은 실제 제작 근거와 사용자 승인에 따라 수정한다.

## 5. 단계별 범위

| 단계 | 제공 범위 | 명시적 제외 |
|---|---|---|
| P0 | 범위·레거시·데이터·golden master·NFR 기준, PostgreSQL·Prisma·서버·CI 기반 | 업무 기능의 대규모 UI 구현 |
| P1 | 인증·권한·감사, 절곡 저장·개정, 계산·전개·3D·DXF 핵심 | 실제 기계 통신 |
| P2 | 기준정보, 수주, 가격, 생산·재단·출력, 데이터 이전·운영 완성 | 실제 기계 통신 |
| P3 | 실제 기계 Agent·전송과 최종 병행 운영·MFC 종료 | 승인되지 않은 신규 외부 연동 |

기계 연동의 1단계 제공 범위는 다음으로 제한한다.

- 비활성 메뉴 또는 화면 항목
- 향후 권한 키
- `planned`, `not-configured` 상태
- 향후 계약 버전의 자리

1단계에는 Windows Agent, COM/TCP/FTP/Bluetooth/전용 DLL 통신, NC 전송 실행, 장비 응답 수집과 현장 시험을 구현하지 않는다.

## 6. 제외 범위

- 입면도 CAD 편집기
- 입면도 템플릿, 분류, 검색, 변수·수식과 절곡 배치
- 입면도 메뉴, API, 활성 Prisma 엔터티와 기능 테스트
- 입면도 출력
- `draww`, `drawxy`, `drawvari`, `selldraw*`의 활성 업무 데이터 변환
- MFC 코드의 기계적인 1:1 웹 변환
- MFC HiStarter, command line DB parameter와 로그인 메커니즘 재사용
- SQLite 업무·테스트·다국어 런타임
- 1차 완료 단계의 다국어 UI·번역 데이터
- 완전한 오프라인 편집
- 외부 ERP·회계 시스템 연동
- 승인되지 않은 추가 기계·프로토콜
- 웹 프로젝트와 직접 관련 없는 회사 시스템 개편

입면도 레거시 데이터의 원본 백업, row count, checksum과 보관 manifest는 포함한다. 이는 입면도 기능 구현이나 활성 데이터 이전이 아니다.

## 7. 용어 기준

| 용어 | 정의 |
|---|---|
| `절곡 문서` | 선·곡선, 절곡 속성, 변수·수식, 재질·계산 설정을 포함하는 웹 도메인 문서 |
| `절곡 템플릿` | 반복 사용하기 위해 분류·개정·게시되는 절곡 문서 |
| `절곡 개정` | 게시 후 변경되지 않는 특정 버전의 절곡 템플릿 |
| `수주·작업` | 거래처·현장과 하나 이상의 절곡 작업 스냅샷을 묶는 업무 집계 단위 |
| `작업 스냅샷` | 수주에 추가한 시점의 절곡 geometry, 변수, 규칙 참조를 보존한 불변 입력 |
| `계산 스냅샷` | 입력 hash, 엔진·규칙 버전과 계산 결과를 함께 보존한 기록 |
| `전개 geometry` | mm 단위 제작 치수의 중립 형상. SVG·DXF·3D의 근거 |
| `입면도` | MFC DrawDx 계열 기능. 레거시 분석·보관만 하며 웹에서는 구현하지 않음 |
| `기계 연동` | 웹 서버, 생산 PC Agent와 NC/시리얼/네트워크/FTP 장비 사이의 실제 전송 기능 |
| `placeholder` | 1단계에 존재하는 비활성 항목·권한·상태·계약 자리. 실제 통신 기능이 아님 |
| `운영 원본` | 현재 실제 업무에서 신규·변경 데이터가 저장되는 권위 있는 레거시 데이터 소스 |
| `golden master` | MFC의 입력·설정·중간값·결과를 고정해 차이를 자동 탐지하는 비교 데이터. MFC 값 자체가 항상 웹의 정답인 것은 아님 |
| `개선 기대값` | MFC 오류·불합리성 또는 승인된 업무 규칙 변경을 반영해 사용자가 승인한 웹의 기대 결과 |
| `활성 이전` | 레거시 데이터를 새 Prisma 업무 엔터티로 변환해 웹 업무에서 사용하는 것 |
| `보관 전용` | 원본과 추적 정보만 보존하며 웹 업무 엔터티로 사용하지 않는 것 |

문서와 UI에서 일반적인 “도면”이라는 단어를 사용할 때 절곡 문서인지 출력물인지 명확히 표시한다. “도면”만으로 입면도까지 포함한다고 해석하지 않는다.

## 9. 검증 원칙

### 9.1 계산과 데이터

1. MFC 입력·중간값·최종값을 먼저 재현해 차이를 가시화한다.
2. 차이를 단순히 웹 오류로 간주하지 않고 다음 유형으로 판정한다.
   - `PARITY_REQUIRED`: 현재 MFC 동작을 유지
   - `LEGACY_DEFECT`: MFC 오류를 수정
   - `WEB_IMPROVEMENT`: 웹 특성에 맞는 흐름·표현 개선
   - `RULE_CHANGE`: 승인된 업무 규칙 변경
   - `UNRESOLVED`: 추가 확인 전 적용 보류
3. `LEGACY_DEFECT`와 `RULE_CHANGE`는 실제 제작 근거, 영향 범위와 사용자 승인 결과를 남긴다.
4. 테스트는 MFC 원본값과 개선 기대값을 구분해 관리한다.

### 9.2 UI와 업무 흐름

1. MFC 화면 배치, 버튼 수와 클릭 순서를 그대로 복제하지 않는다.
2. 웹의 자동 저장, 검색, 이력, 명확한 오류 안내, 반응형 조회와 동시 접근 제어를 적극 활용한다.
3. 동일한 업무 목적을 정확하고 더 편리하게 완료하는지를 검증한다.
4. 업무 완료율, 입력 오류, 작업 시간, 데이터 누락과 사용자 승인을 검증 기준으로 사용한다.

## 10. 범위 변경 규칙

1. 요청을 기존 작업의 결함, 명세 보완 또는 범위 추가로 분류한다.
2. 범위 추가라면 사용자 가치, 선행 의존성, 데이터·API·UI·운영 영향을 기록한다.
3. 현재 작업과 게이트 일정에 미치는 영향을 분석한다.
4. 프로젝트 책임자가 포함·보류·별도 프로젝트 중 하나를 결정한다.
5. 승인된 변경만 작업 ID, 종합 설계와 작업계획서에 반영한다.
6. 입면도 기능 요청은 항상 범위 변경으로 처리한다.
7. P0~P2에서 실제 기계 통신 요청이 나오면 P3 작업으로 분리한다.
